Aulas 01 & 02
Os 4 Pilares da Orientação a Objetos
01 — PILAR
Abstração
Modelar apenas as características essenciais de um objeto, ignorando detalhes irrelevantes para o sistema.
Exemplo Conta Bancária: modela saldo, titular, depositar(), sacar().
Ignora: cor do cartão, material do cofre.
02 — PILAR
Herança
Uma subclasse (filha) herda automaticamente atributos e métodos de uma superclasse (pai), podendo adicionar ou sobrepor comportamentos.
Exemplo Pessoa → PessoaFísica (+ cpf), PessoaJurídica (+ cnpj).
Ambas herdam nome de Pessoa.
03 — PILAR
Encapsulamento
Reunir dados e métodos dentro de uma classe, controlando o acesso externo com modificadores de visibilidade (público/privado).
Exemplo Atributo saldo é private.
Só pode ser acessado via getSaldo() (método público).
04 — PILAR
Polimorfismo
Objetos de classes diferentes respondem de formas distintas à mesma mensagem. "Muitas formas" para a mesma operação.
Exemplo Mensagem "Fale!":
Cachorro → "Au au!" · Gato → "Miau!" · Pato → "Quack!"

📦

Diagrama de Classes

Mostra as classes do sistema, seus atributos, métodos e os relacionamentos entre elas (associação, herança, composição, agregação). É o diagrama mais usado em UML.

🎯

Diagrama de Casos de Uso

Mostra o que o sistema faz do ponto de vista do usuário. Representa atores, casos de uso e seus relacionamentos (include, extend, generalização).

🔁

Diagrama de Sequência

Mostra a ordem das mensagens trocadas entre objetos para executar um caso de uso. Eixo X = objetos, Eixo Y = tempo (de cima para baixo).

🔄

Diagrama de Atividades

Similar a um fluxograma, mas com swimlanes para separar responsabilidades. Modela o fluxo de trabalho (workflow) de um processo ou algoritmo.

⚡

Diagrama de Máquina de Estados

Modela os estados de um objeto e as transições entre eles disparadas por eventos. Ex: Pedido: Pendente → Aprovado → Enviado → Entregue.

🏗️

Diagrama de Implantação

Descreve como o software é distribuído sobre o hardware. Mostra nós (servidores, dispositivos) e as conexões entre eles.

Aula 03
BPMN — Notação para Modelagem de Processos
📌 O que é BPMN? Business Process Model and Notation — linguagem visual padronizada para representar processos de negócio, mantida pelo OMG. Usa símbolos padronizados que qualquer pessoa do negócio ou de TI consegue entender.
Evento de Início
Círculo borda fina — onde o processo começa
Evento Intermediário
Círculo duplo — ocorre no meio do processo
Evento de Fim
Círculo borda grossa — onde o processo termina
Tarefa (Atividade)
Retângulo arredondado — passo do processo
Gateway XOR
Losango com X — apenas UM caminho (exclusivo)
Gateway AND
Losango com + — TODOS os caminhos obrigatórios
Gateway OR
Losango com círculo — um ou mais caminhos
Evento Mensagem
Círculo com envelope — disparado por mensagem
Evento de Tempo
Círculo com relógio — início em hora/data
Pool
Retângulo externo — participante do processo
Lane
Divisão do Pool — setor, cargo ou departamento
Subprocesso
Retângulo com + — processo dentro de processo
Exemplo BPMN — Pedido de Pizza (simplificado)

AS IS — Como é Agora

Documentação da situação atual do processo. Identifica problemas, fragilidades e oportunidades. Feito junto aos gestores com entrevistas, observação ou reunião JAD.

TO BE — Como Será

Situação futura melhorada. Pense em como o processo DEVERIA ser, sem se preocupar com restrições. Simplifique ao máximo e valide com a alta gestão.

Aula 04
Regras de Negócio — Identificação e Documentação
TIPO 01

Termos

Elementos básicos da linguagem do negócio. Definir um termo já é uma regra. Ex: o que significa "cliente ativo" na empresa.

TIPO 02

Fatos

Descrevem a natureza do negócio. Ex: "Um cliente pode ter vários pedidos."

TIPO 03

Cálculos / Derivações

Fórmulas de transformação. Ex: desconto calculado com base na quantidade comprada.

TIPO 04

Restrições

Restringem comportamentos. Ex: "Pedido só pode ser cancelado se ainda não foi enviado."

TIPO 05

Habilitações de Ação

Formato SE → ENTÃO. Ex: SE cliente atrasar 3 meses, ENTÃO bloquear novos pedidos.

Atributos de Qualidade
🎯
Atomicidade
A responsabilidade da regra deve ser única e indivisível. Uma regra = uma responsabilidade.
🔍
Verificável
Deve ser possível testar ou mensurar se a regra foi corretamente implementada no sistema.
🔗
Rastreável
Ao final do projeto, deve ser possível identificar onde a regra está sendo atendida no produto.
✅
Consistência
Não pode contradizer outra regra nem estimular ações contraditórias entre si.
💡 RN × Requisitos de Software Regras de Negócio definem como o negócio funciona — elas não têm prioridade.
Requisitos de Software são como o sistema atenderá o negócio — esses sim recebem prioridade do cliente.
As RNs geram os requisitos de software.
